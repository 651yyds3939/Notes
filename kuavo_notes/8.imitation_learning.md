# 📖 Kuavo 机器人比赛：部署与排坑全纪录 (终极完整版) [cite: 1]

## 🌟 第零部分：生存铁律（Git与Docker的正确食用姿势） [cite: 1]

在我们开始折腾复杂的机器人之前，必须先牢记这两条底层操作铁律。 [cite: 1]

### 1. 单份代码库切换分支的“必做仪式” [cite: 1]
当你仅有一份 `kuavo_data_challenge` 本地仓库，且需要在不同比赛分支（如 **Tianchi** 与 **ICRA**）间频繁切换时，必须严格执行以下流程： [cite: 1]

#### 🛠️ 第一步：强力清场（净身出户） [cite: 1]
在切换前，必须彻底清除由于本地调试、脚本自动修改（如 `sed` 全局替换）产生的改动。 [cite: 1]

```bash
cd ~/kuavo_data_challenge

# 1. 放弃所有受版本控制文件的修改（恢复官方原始状态）
git checkout .

# 2. 彻底移除所有本地未跟踪的临时文件/编译产物
git clean -fd
```

> **为什么要清场？**
> 调试时我们经常会执行“暴力手术”（如删除 `.launch` 里的绑核代码或修改 Python 补丁）。如果不清除，这些“毒代码”会跟随 `checkout` 被带入新分支，引发严重的 Git 冲突或不可预知的逻辑错误。 [cite: 1]

#### 🗺️ 第二步：环境确认 [cite: 1]
在盲目切换前，先确认你当前所在的位置以及目标分支是否存在。 [cite: 1]

```bash
# 查看本地及远程所有分支
git branch -a
# 注意：带 * 号的是你当前所在的分支
```

#### 🚀 第三步：精准跳跃 [cite: 1]
执行分支切换动作。 [cite: 1]

```bash
# 切换到目标分支
git checkout <目标分支名>

# 示例：
# git checkout main
# git checkout icra_challenge
```

#### 🔗 第四步：同步外包（核心补丁） [cite: 1]
这是最容易被遗忘、也是最致命的一步。机器人项目依赖大量的 3D 模型和底层库，它们通常以 **Submodule** 形式存在。 [cite: 1]

```bash
# 强制同步子模块，确保模型文件（.xml）与当前代码分支配套
git submodule update --init --recursive
```

> **如果不做这一步：**
> 经常会出现“找不到 `scene1.xml` 模型”或“模型版本不匹配”的报错，因为代码逻辑更新了，但存放模型的子文件夹还停留在旧分支的状态。 [cite: 1]

### 2. Docker 启动与管理指南 [cite: 1]
我们的代码是存放在 Ubuntu 宿主机上的，而 Docker 只是一个“借用”了宿主机代码的纯净运行环境（类似于插拔式的系统U盘）。 [cite: 1]
* **启动并进入新 Docker 的标准命令（每次开机必敲）：** [cite: 1]
  ```bash
  cd ~/kuavo-ros-opensource
  sudo bash docker/run_with_gpu.sh kuavo_opensource_mpc_wbc_img:0.6.1
  ```
* **关于“如何修改 Docker”的认知误区：** [cite: 1]
  不要试图把修改环境的操作（比如 `apt install psmisc`）永久保存在这个 Docker 里。因为我们的架构是把宿主机的代码文件夹直接**挂载（映射）**进了容器。 [cite: 1]
  **正确姿势**：在 Ubuntu 宿主机用 VSCode 改代码，Docker 里面会**瞬间同步生效**。Docker 只是个无情的执行机器，用完即焚，下次启动依然是纯净的 0.6.1 环境。 [cite: 1]

## 🏗️ 第一部分：核心架构理念——“大脑与身体分离” [cite: 1]
在我们部署的最终方案中，彻底抛弃了在同一个环境里配齐所有依赖的幻想，采用了真实的工业级部署架构： [cite: 1]
* **“大脑”（推理端）**：在 **Ubuntu 宿主机** 的 Conda 环境（`kdc`, Python 3.10）中运行。这里算力最强、环境最纯净，专门负责加载最新的 AI 大模型（Lerobot 0.4.2）。 [cite: 1]
* **“身体”（仿真端）**：在 **Docker 容器**（`0.6.1` 镜像, Python 3.8）中运行。这里专门利用官方打包好的 ROS Noetic 和底层 C++ 依赖库，用来渲染 MuJoCo 画面和执行底层电机控制逻辑。两者通过网络进行通信。 [cite: 1]

## 🛑 第二部分：仿真端（身体）踩坑录 [cite: 1]
我们在拉起 MuJoCo 仿真器时，遭遇了官方代码里的“连环雷”： [cite: 1]
https://github.com/LejuRobotics/kuavo-ros-opensource/blob/opensource/kuavo-data-challenge/readme.md [cite: 1]

1. **跨服聊天之坑（ICRA 与 Tianchi 的混淆）** [cite: 1]
   * **现象**：看着天池（Tianchi）比赛的教程，却在跑 ICRA 挑战赛的脚本，导致各种路径不对。 [cite: 1]
   * **解决**：理清目录结构，天池的仿真入口在 `data_challenge_simulator` 目录下，而 ICRA 的在 `kuavo_deploy` 下。 [cite: 1]
2. **“富贵病”绑核之坑（最隐蔽的致命 Bug）** [cite: 1]
   * **现象**：MuJoCo 闪退，报错 `REQUIRED process [ekf_se-12] has died` 和 `taskset: Invalid argument`。 [cite: 1]
   * **原因**：官方把状态估计节点（`ekf_se`）硬编码绑定在了第 18 号 CPU 核心上（`taskset -c 18`）。普通笔记本（如 8 核 16 线程）根本没有 18 号核心，导致系统拒绝执行并触发 ROS 连环自毁。 [cite: 1]
   * **解决**：使用 `sed` 命令全局搜索并删除了所有 `.launch` 文件里的 `taskset` 绑核指令。 [cite: 1]
3. **僵尸节点抢注之坑** [cite: 1]
   * **现象**：报错 `new node registered with same name`，仿真器秒退。 [cite: 1]
   * **原因**：之前的崩溃导致后台残留了没死透的 `roscore` 等进程，新节点启动时发生名字冲突。 [cite: 1]
   * **解决**：一键超度（`pkill -9 roscore` / `killall`），保持跑道干净。 [cite: 1]
4. **版本错位之坑（找不到 xml 文件）** [cite: 1]
   * **现象**：报错 `could not open file .../biped_s42/xml/scene1.xml`。 [cite: 1]
   * **原因**：代码里写死了要找 `s42`（夸父4.2版）的模型文件，但官方代码库里实际装的是 `s45`（4.5版）的文件。 [cite: 1]
   * **解决**：使用软链接（`ln -s biped_s45 biped_s42`），用一招“移花接木”骗过了代码。 [cite: 1]

## 🧠 第三部分：推理端（大脑）与模仿学习探秘 [cite: 1]
这部分揭示了模型到底是怎么来的，以及为何代码跑不通。 [cite: 1]
https://github.com/LejuRobotics/kuavo_data_challenge/tree/main [cite: 1]

### 1. 模仿学习（Imitation Learning）中核心文件的作用： [cite: 1]
* **`.bag` 文件（原材料 / 录像带）**：Hugging Face 网站上放的全是这个。它是人类工程师拿着手柄遥控机器人抓东西时，记录下的全部传感器和动作录像。 [cite: 1]
* **`parquet` 文件（半成品食材）**：将 `.bag` 转换后的格式，专门喂给神经网络进行训练。 [cite: 1]
* **`train_policy.py`（炼丹炉）**：执行这段代码，显卡就会开始学习 `.bag` 里的录像，试图模仿人类的动作。 [cite: 1]
* **`config.json` 与 `model.safetensors`（成品大脑 / 权重）**：训练几万步之后保存下来的最终文件。推理时只要有这两个文件，机器人就知道该怎么动了。 [cite: 1]

### 2. 推理端踩坑录： [cite: 1]
* **结构代差之坑**：强行用旧版的模型权重去匹配最新的 `lerobot 0.4.2` 代码，导致疯狂报错 `Missing keys`（缺少 `model`、`net` 层级和 `depth_encoder`）。 [cite: 1]
* **Hugging Face 幻觉之坑**：以为官方在网上放了“成品大脑”，解开协议权限后才发现，官方只给了 `.bag` 原材料，本意是让参赛者自己拿显卡去炼丹。 [cite: 1]
* **最终对策**：不拿笔记本熬夜炼丹（容易爆显存和内存），直接向队友索要他们已经训练好的、包含了 `config.json` 和 `model.safetensors` 的满血 `outputs` 文件夹。 [cite: 1]

## 💻 第四部分：打通任督二脉（最终部署连招） [cite: 1]

当你拿到队友给的模型文件（假设叫 `my_model` 文件夹），按以下顺序分三个终端执行，即可完美跑通： [cite: 1]

### 【终端 1】：恢复代码纯净（在 Ubuntu 宿主机） [cite: 1]
每次开工前的第一件事： [cite: 1]
```bash
cd ~/kuavo_data_challenge
git checkout .
```

### 【终端 2】：拉起“身体”（在 Docker 容器内） [cite: 1]
启动仿真器并扫清底层雷区： [cite: 1]
```bash
# 1. 启动并进入 Docker
cd ~/kuavo-ros-opensource
sudo bash docker/run_with_gpu.sh kuavo_opensource_mpc_wbc_img:0.6.1

# 2. 清理残余僵尸进程
pkill -9 roscore
pkill -9 rosmaster
pkill -9 roslaunch
pkill -9 python
pkill -9 python3

# 3. 解决 CPU 绑核 Bug (只需执行一次)
cd /root/kuavo_ws
find src/ -name "*.launch" -exec sed -i 's/launch-prefix="taskset -c [0-9]*"//g' {} +

# 4. 解决模型版本路径错位 (只需执行一次)
cd /root/kuavo_ws/src/data_challenge_simulator/models/
ln -s biped_s45 biped_s42

# 5. 启动 MuJoCo 仿真器 (Tianchi 天池版)
cd /root/kuavo_ws/src/data_challenge_simulator/examples/deploy
python3 deploy.py
# 输入任务编号（如 1），回车，等待 MuJoCo 画面稳定。
```

### 【终端 3】：唤醒“大脑”（在 Ubuntu 宿主机） [cite: 1]
MuJoCo 跑起来后，在宿主机发射动作指令： [cite: 1]
```bash
# 1. 激活 3.10 AI 环境
conda activate kdc

# 2. (非常重要) 去修改你的配置文件 configs/deploy/kuavo_sim_env.yaml 
# 把 pretrained_path 改成你找队友拿到的模型的绝对路径，千万不要带文件后缀！
# 示例：pretrained_path: "/home/lwy/kuavo_data_challenge/my_model"

# 3. 运行推理测试，给机器人下达控制指令
cd ~/kuavo_data_challenge
python kuavo_deploy/src/scripts/script_auto_test.py --config configs/deploy/kuavo_sim_env.yaml
```
